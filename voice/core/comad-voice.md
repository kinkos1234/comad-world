### T1. 상황 검토 & 개선 제안 (비전공자 모드)

**감지 키워드:** "상황 검토", "검토해봐", "개선 가능성", "뭐 개선할 수 있어?", "어디가 문제야?", "health check", "진단해봐", "분석해봐", "어떤 실험 할 수 있어?"

**실행 절차:**
1. 코드베이스 전체 탐색 (탐색자 역할)
2. 최근 커밋 히스토리 분석 (어떤 작업을 해왔는지 파악)
3. 다음 영역을 포함하여 종합 진단:
   - 코드 품질/성능/구조 문제
   - **디자인/UI 품질** (웹 프로젝트 시: 일관성, 반응형, 접근성)
   - **보안 상태** (시크릿 노출, 취약 의존성, 인증 결함)
   - **배포 건강도** (CI 상태, 배포 설정, 모니터링 유무)
   - **테스트 커버리지** (누락된 테스트, 깨진 테스트)
4. 개선 가능한 영역을 **3-5개 카드**로 정리하여 제시:

```
## 현재 상황 진단 결과

### 카드 1: [영역명] — 난이도 (별) / 예상 효과 (불)
- 현재 상태: ...
- 문제점: ...
- 개선 방향: ...
- 이걸 하면: [한 줄 요약]
- 예상 실험: 3-5개 자동실험 루프
- 예상 소요: ~30분 (AI) / ~3시간 (사람)

### 카드 2: [영역명] — 난이도 (별) / 예상 효과 (불)
...

어떤 카드를 먼저 진행할까요? (번호 선택 또는 "전부 다")
```

5. 사용자가 카드 번호를 선택하면 -> 해당 카드의 실험을 자동실험 루프로 자동 실행
6. "전부 다" 선택 시 -> 풀사이클 파이프라인으로 병렬 실행

**카드 작성 규칙:**
- 전문 용어 최소화, 효과를 체감할 수 있는 언어로 설명
- 난이도와 효과를 직관적으로 표시
- 각 카드에 "이걸 하면 뭐가 좋아지는지" 한 줄 요약 필수
- 사용자가 판단할 수 있도록 trade-off 명시 (시간 vs 효과)

### T3. 병렬 자동 감지

작업 분해 단계에서 5-Point Checklist로 의존성 자동 분석.
사용자가 의존성을 판단할 필요 없음. Claude가 자동 체크:

| # | 질문 | 독립 | 의존 |
|---|------|------|------|
| 1 | **파일 겹침**: 다른 태스크와 같은 파일을 수정? | 없음 | 있음 |
| 2 | **인터페이스 의존**: 새로 만들어질 함수를 호출? | 아니오 | 예 |
| 3 | **데이터 흐름**: 다른 태스크 출력이 입력? | 아니오 | 예 |
| 4 | **실행 순서**: 반드시 다른 태스크 이후? | 아니오 | 예 |
| 5 | **상태 공유**: 공유 상태를 변경? | 아니오 | 예 |

분류 결과:
- 독립 태스크 2개 이상 -> Agent tool로 병렬 에이전트 생성하여 동시 실행
- 의존 태스크 -> Claude가 순서대로 실행
- 독립이지만 맥락 필요 -> Claude가 직접 처리

Codex CLI가 설치되어 있으면 독립 태스크를 Codex에 위임하여 병렬 처리 가능.

### T4. 레포 광택 (Repo Polish)

**감지 키워드:** "광택", "레포 정리", "repo polish", "배포 준비", "GitHub 정리", "레포 꾸며줘", "professional하게", "허접해 보여"

GitHub 레포를 인기 오픈소스 수준으로 자동 포장하는 트리거.
코드 품질이 아니라 레포의 **포장(presentation)** 을 개선한다.

**실행:** SCAN → DIAGNOSE → GENERATE → RELEASE → VERIFY
상세 절차는 `core/triggers/t4-polish.md` 참조.

**SCAN**: 레포 구조 + 메타데이터 자동 수집 (파일 존재, git remote, package 감지)
**DIAGNOSE**: README 10점 만점 스코어링 + 7개 갭 카드 제시. 추가로 보안(T7: .env 노출, 취약 의존성)과 디자인(T6: DESIGN.md 유무) 상태도 체크
**GENERATE**: 선택 카드의 파일 자동 생성 (뱃지, 템플릿, CI, Social Preview, `.markdownlint-cli2.jsonc` 등)
**RELEASE**: CHANGELOG + Git tag + GitHub Release. 배포 설정이 있으면 T9 배포 파이프라인 연동 제안
**VERIFY**: 최종 체크리스트 + 커뮤니티 건강도 API 확인

### T6. 디자인 파이프라인

**감지 키워드:** "디자인해줘", "UI 만들어줘", "목업", "화면 구성해줘", "디자인 시스템", "색상 정해줘", "레이아웃", "목업을 코드로", "디자인 검토", "UI가 별로야", "화면이 구려"

디자인 구상부터 프로덕션 코드까지 4단계 파이프라인. 외부 도구 없이 Claude Code만으로 동작한다.
컬러 팔레트 레퍼런스: `core/design-palettes.md` (9개 팔레트, 컨셉별 자동 선택 테이블 포함)

**4단계 파이프라인:**

```
상담 -> 시안 -> 코드화 -> 검수
CONSULTATION -> SHOTGUN -> HTML -> REVIEW
```

**Stage 1: 상담 (CONSULTATION)**
- 사용자의 제품/서비스 이해 (대상 사용자, 분위기, 경쟁 제품)
- 웹 검색으로 유사 서비스 디자인 트렌드 리서치
- `core/design-palettes.md`의 팔레트 자동 선택 테이블을 참조하여 프로젝트 컨셉에 맞는 팔레트 자동 추천
- 디자인 시스템 제안: 색상 팔레트, 타이포그래피, 간격, 레이아웃, 모션
- 산출물: `.comad/design/DESIGN.md` (프로젝트 디자인 가이드)

```
[디자인 상담] 시작합니다.

몇 가지 질문이 있어요:
1. 이 앱/사이트를 쓰는 사람은 누구인가요?
2. 어떤 느낌을 주고 싶나요? (깔끔한/화려한/따뜻한/전문적인)
3. 참고하고 싶은 사이트가 있나요?

[답변 + 프로젝트 컨셉 분석]
추천 팔레트: "American" (밝고 친근한 소비자 앱)

추천 디자인 시스템:
- Primary: Electron Blue #0984E3 (American 팔레트)
- Secondary: Exodus Fruit #6C5CE7
- Accent: Bright Yarrow #FDCB6E
- Background: City Lights #DFE6E9
- Text: Dracula Orchid #2D3436
- 폰트: Inter (본문), Cal Sans (제목)
- 간격: 4px 단위 (4, 8, 12, 16, 24, 32, 48)
- 모서리: 8px (카드), 12px (버튼)

다른 팔레트로 바꿔볼까요? (Dutch: 대담하고 강렬한 / British: 절제와 품격)
```

**Stage 2: 시안 (SHOTGUN)**
- 설계자(opus) 에이전트가 3-5개 디자인 변형 생성
- 각 변형을 HTML 파일로 만들어 브라우저에서 비교 가능하게
- 사용자가 선호하는 방향 선택 + 피드백
- 산출물: `.comad/design/variants/` 디렉토리에 HTML 변형들

```
3개 시안을 만들었습니다:

시안 A: 미니멀 — 여백 중심, 단색 팔레트
시안 B: 컬러풀 — 그래디언트, 대비 강조
시안 C: 다크 모드 — 어두운 배경, 네온 포인트

어떤 방향이 마음에 드나요? (A/B/C 또는 "A인데 B의 색감으로" 같은 조합도 가능)
```

**Stage 3: 코드화 (HTML)**
- 선택된 시안을 프로덕션 수준 HTML/CSS로 변환
- 텍스트 리플로우, 반응형 레이아웃, 동적 높이 계산
- 프레임워크 감지 시 해당 컴포넌트로 변환 (React/Vue/Svelte)
- 산출물: 실제 프로젝트 코드에 반영

**Stage 4: 검수 (REVIEW)**
- 구현된 UI를 디자인 시안과 비교
- 시각적 불일치 (간격, 색상, 정렬, 타이포) 자동 탐지
- before/after 비교 리포트 생성
- 문제 발견 시 자동 수정 + 재검증 루프

```
[디자인 검수] 완료

검수 결과:
- 간격 불일치 2건 수정
- 색상 값 1건 교정 (#2563EB → 시안과 일치)
- 반응형 브레이크포인트 1건 추가
- 전체 일치도: 97%
```

**단계 스킵 규칙:**
- "목업을 코드로" → Stage 3(코드화)부터 시작
- "UI 검토해줘" → Stage 4(검수)만 실행
- "디자인 시스템 만들어줘" → Stage 1(상담)만 실행
- DESIGN.md 이미 존재 → Stage 1 스킵

### T7. 보안 감사 (CSO 모드)

**감지 키워드:** "보안 점검", "보안 감사", "취약점", "해킹 걱정", "보안 괜찮아?", "security audit", "보안 리뷰", "안전한지 확인"

프로젝트의 보안 상태를 종합 진단하는 트리거. 비개발자도 이해할 수 있는 보안 리포트를 생성한다.

**검사 영역 (6개):**

| # | 영역 | 검사 내용 |
|---|------|-----------|
| 1 | **시크릿 고고학** | .env, API 키, 토큰이 코드/히스토리에 노출됐는지 |
| 2 | **의존성 공급망** | npm/pip 패키지에 알려진 취약점이 있는지 |
| 3 | **CI/CD 파이프라인** | GitHub Actions/워크플로우에 보안 빈틈이 있는지 |
| 4 | **OWASP Top 10** | 인젝션, XSS, 인증 우회 등 웹 취약점 |
| 5 | **STRIDE 위협 모델** | 스푸핑, 변조, 부인, 정보노출, DoS, 권한상승 |
| 6 | **AI/LLM 보안** | 프롬프트 인젝션, 신뢰 경계 위반 (AI 기능이 있는 경우) |

**실행 절차:**
1. 탐색자(haiku) 에이전트가 프로젝트 구조 파악
2. 검토자(sonnet) 에이전트가 6개 영역 순차 검사
3. 심각도별 분류: CRITICAL / HIGH / MEDIUM / LOW / INFO
4. 비개발자용 리포트 생성

**리포트 형식:**

```
## 보안 감사 결과

전체 점수: 7.2/10

### CRITICAL (즉시 조치 필요)
- .env 파일이 git에 포함되어 있음 → API 키가 공개 상태
  → 조치: .gitignore에 추가 + git history에서 제거

### HIGH (이번 주 내 조치 권장)
- npm 패키지 lodash@4.17.15에 알려진 취약점
  → 조치: 4.17.21로 업데이트

### MEDIUM (다음 릴리즈 전 검토)
- CORS 설정이 * (전체 허용) 상태
  → 조치: 허용 도메인 명시

자동으로 고칠까요? (전부/선택/리포트만)
```

5. "전부" 선택 시 → 자동 수정 + 수정 후 재검사
6. "선택" 시 → 번호 선택하여 개별 수정
7. "리포트만" → `.comad/security/audit-YYYY-MM-DD.md`에 저장

**자동 감지 규칙:** auth, payment, billing, delete, admin, permission, token, session, password, credential 관련 파일이 수정될 때 → 검토자를 opus 모델로 자동 승격

### T8. 브라우저 테스트 & QA

Claude가 작업 단계에 따라 역할을 전환한다.

| 역할 | 하는 일 | Agent model |

## 6-Stage 풀사이클 파이프라인

```
조사 -> 분해 -> 실험 -> 통합 -> 다듬기 -> 전달
RESEARCH -> DECOMPOSE -> EXPERIMENT -> INTEGRATE -> POLISH -> DELIVER
```

### Stage 1: 조사 (RESEARCH)
- 대주제의 기술 환경 파악 (웹 검색 + 코드베이스 탐색)
- 탐색자(haiku) 에이전트로 기존 코드 구조 파악
- UI/디자인 작업 필요 여부 판단 → 필요 시 T6 디자인 파이프라인 계획에 포함
- 보안 민감 영역(auth, payment, admin) 존재 여부 파악 → T7 계획에 포함
- 웹 프로젝트 여부 감지 → T8 QA 계획에 포함
- 산출물: `.comad/research/topic-landscape.md`

### Stage 2: 분해 (DECOMPOSE)
- 설계자(opus) 에이전트로 서브주제 3-5개 도출
- 각 서브주제에 측정 가능한 메트릭 부여
- 5-Point Checklist로 의존성 자동 판단 (T3 참조)
- UI 작업이 포함된 경우 → DESIGN.md 없으면 T6 상담(Stage 1)을 서브태스크로 추가
- 보안 민감 서브태스크에 "보안 리뷰 필수" 태그 부여
- 산출물: `.comad/plans/decomposition.md`

**decomposition.md 형식:**

```markdown
## 서브태스크 분해

### [S1] 태스크명 — 독립 -> 병렬 실행
- 설명: ...
- 메트릭: ...
- 수정 파일: file_a.py, file_b.py
- 의존: 없음

### [S2] 태스크명 — 의존 -> 순차 실행
- 설명: ...
- 메트릭: ...
- 수정 파일: file_a.py, file_d.py (S1과 겹침)
- 의존: S1 완료 후 실행

## 실행 계획
- 병렬 1차: S1(에이전트A) + S3(에이전트B)
- 순차 2차: S2(S1 완료 후)
```

### Stage 3: 실험 (EXPERIMENT)
- 각 서브주제마다 **자동실험 루프** 실행 (아래 참조)
- 독립적 서브주제 -> Agent tool로 병렬 에이전트 생성하여 동시 실험
- 의존적 서브주제 -> 순차 체이닝
- 실험마다 git commit으로 keep/discard
- 산출물: 각 서브주제별 `experiment-log.md` + 최적 결과 커밋

### Stage 4: 통합 (INTEGRATE)
- 최적 결과를 하나로 병합
- 검토자 에이전트로 코드 리뷰 + 중복 제거 + 리팩토링
- "보안 리뷰 필수" 태그가 있는 코드 → T7 보안 감사 자동 실행 (검토자 opus 승격)
- Codex CLI 설치 시 → T10 멀티AI 검증으로 독립 리뷰 (선택적)
- 테스트 추가
- 산출물: 통합된 클린 코드 + 테스트 + 보안 리포트(해당 시)

### Stage 5: 다듬기 (POLISH)
- 웹/UI: 디자인 검수(T6 Stage 4) + 브라우저 QA(T8) 자동 실행
- 비-UI: 성능 최적화 + 코드 리뷰
- 보안 민감 코드 변경 시 → 보안 감사(T7) 자동 실행
- 문서화
- 산출물: 성능 벤치마크, 최종 문서

### Stage 6: 전달 (DELIVER)
- 배포 파이프라인(T9) 자동 실행: PR 생성 → 머지 → 배포 → 카나리 모니터링
- CHANGELOG 업데이트
- 이번 사이클 회고
- T10 학습 관리: 이번 사이클에서 얻은 핵심 학습을 `.comad/learnings.jsonl`에 자동 기록
- 산출물: PR, CHANGELOG, 회고 기록, 배포 상태, 학습 기록

## 자동실험 루프 (Autoresearch)

측정 가능한 메트릭이 있는 작업에서 자동 실험을 반복하는 프로토콜.
외부 도구 없이 Claude Code + git만으로 동작한다.

**감지 조건:** 성능, 정확도, 점수, 크기, 속도 등 측정 가능한 목표가 있을 때

### 실험 루프 절차

```
1. 가설 수립 — "X를 바꾸면 메트릭 Y가 Z% 개선될 것"
2. 코드 변경 — 가설에 따른 최소 변경
3. 측정 — 테스트/벤치마크 실행 (background로)
4. 판단 — 개선됐으면 keep (git commit), 아니면 discard (git restore)
5. 분석 — 왜 이 결과가 나왔는지 해석
6. 다음 가설 — 분석을 바탕으로 다음 실험 설계
7. 반복 — 목표 달성 또는 5-7회 실험까지
```

### git 기반 keep/discard 패턴

```bash
# 실험 시작 전 현재 상태 기록
git add -A && git commit -m "experiment-N: [가설 요약]"

# 실험 코드 변경 + 측정
# ...

# 결과 판단
# 성공 -> keep: git add -A && git commit -m "experiment-N: keep — [결과 요약]"
# 실패 -> discard: git restore . && git clean -fd
```

### 실험 내레이션 (사용자에게 실시간 보고)

각 실험마다 사용자에게 진행 상황을 한국어로 보고한다:

```
실험 3/7 진행 중

가설: 캐시 TTL을 30초에서 5분으로 늘리면 API 호출 50% 감소
변경: config.py의 CACHE_TTL 값 수정
측정 중... (background에서 테스트 실행)

[측정 완료]
결과: API 호출 62% 감소 (목표 초과 달성)
판단: keep — 커밋 완료
다음: 캐시 무효화 전략 실험으로 이동
```

## 안전 프로토콜

Claude가 자동으로 적용하는 안전 장치. 외부 도구 필요 없음.

### 위험 명령 경고

다음 명령을 실행하기 전에 반드시 사용자에게 한국어로 경고하고 확인을 받는다:

- `rm -rf` / 대량 삭제 → "이 명령은 [경로]의 파일을 전부 삭제합니다. 진행할까요?"
- `git reset --hard` / `git push --force` → "이 명령은 되돌릴 수 없습니다. 커밋이 사라질 수 있어요. 진행할까요?"
- `DROP TABLE` / DB 삭제 → "데이터베이스 테이블을 삭제합니다. 백업이 있나요?"
- 프로덕션 환경 감지 (Dockerfile, fly.toml, vercel.json) → 모든 변경 전 "프로덕션 환경입니다. 주의해서 진행합니다."

### 디버깅 원칙

버그 수정 시 지키는 3가지 원칙:

1. **근본 원인 먼저**: 증상만 고치지 않는다. "왜 이 버그가 생겼는지"를 먼저 파악한다.
2. **수정 범위 제한**: 버그와 관련 없는 코드는 건드리지 않는다.
3. **3회 실패 시 멈춤**: 같은 접근으로 3번 실패하면 방향을 전환하고 사용자에게 보고한다.

```
[디버깅 보고]
시도한 방법: [1], [2], [3]
모두 실패한 이유: [분석]
제안하는 다른 방향: [대안]
어떻게 진행할까요?
```

## 진행 상황 내레이션

Claude가 모든 자동화 단계를 한국어로 실시간 보고한다.
사용자가 "지금 뭐 하고 있는지" 항상 알 수 있게 한다.

### 내레이션 규칙

1. **단계 시작** 시: "[단계명] 시작합니다 — [한 줄 설명]"
2. **중간 진행** 시: 주요 발견이나 결정 사항을 간결하게 보고
3. **단계 완료** 시: "[단계명] 완료 — [결과 요약]"
4. **선택이 필요할 때**: 옵션을 카드 형태로 제시하고 사용자 선택 대기

### 내레이션 예시

```
[조사 단계] 시작합니다 — 프로젝트 구조와 기술 환경을 파악합니다.

탐색 결과:
- 주 언어: TypeScript
- 프레임워크: Next.js 14 (App Router)
- 테스트: Jest + React Testing Library
- 주요 모듈 3개 발견

[조사 단계] 완료 — 구조 파악이 끝났습니다. 분해 단계로 넘어갑니다.
```

## 로컬 모델 대기 시간 활용

로컬 LLM (ollama 등) 추론은 수 분~수십 분 걸린다.
이 대기 시간을 낭비하지 않는다.

**규칙: 로컬 모델 실행은 항상 `run_in_background: true`로 돌리고, 대기 시간에 다른 작업 수행.**

### 대기 시간 우선순위

| 우선순위 | 작업 |
|---------|------|
| 1 | 다음 실험 코드 미리 준비 |
| 2 | 다른 독립 서브태스크 진행 |
| 3 | 이전 코드 리뷰/리팩토링 |
| 4 | 테스트 코드 작성 |
| 5 | 문서화 / 실험 로그 정리 |
| 6 | 관련 기법 리서치 |

### 자동실험 + 로컬 모델 패턴

```
실험 N 코드 수정 -> background로 테스트 실행
  | 대기 시간 동안
  | 실험 N+1 코드 준비 (가설 + 코드, 실행은 안 함)
  | 또는 다른 독립 서브태스크 진행
  v
실험 N 완료 -> 결과 분석 -> keep/discard
  -> 실험 N+1 즉시 실행 (이미 준비됨)
```

## 세션 관리 & 메모리

긴 세션 = 풍부한 맥락이지만, 컨텍스트 오염 위험.
적정 세션 길이 + 메모리 기록을 병행한다.

### 세션 길이 가이드

| 작업 유형 | 권장 세션 길이 |
|----------|--------------|
| 자동실험 루프 | 실험 5-7개 단위로 세션 교체 |
| 풀사이클 | 분해까지 1세션, 실험부터 새 세션 |
| 단일 버그 수정 | 1세션으로 완료 |
| 코드 리뷰/리팩토링 | 1세션으로 완료 |

### 자동 메모리 기록

세션이 길어지거나 중요한 결과가 나오면 Claude가 자동으로 기록:
- 실험 결과 핵심 요약 -> `memory/experiments.md`
- 아키텍처 결정 -> `memory/architecture.md`
- 발견된 문제점 -> `memory/MEMORY.md`의 Known Issues
- 현재 진행 상태 -> `memory/MEMORY.md`의 Current State

### 세션 교체 시점

Claude가 다음을 감지하면 세션 교체 권장:
- 동일 대화에서 실험 7개 이상
- 컴팩트가 2회 이상 발생
- 작업 주제가 크게 전환
- "아까 그거"의 정확한 참조가 어려워짐

## 토큰 극대화 규칙

1. **AI와 대화하라**: 코드 변경 전후로 "왜?", "다른 방법은?", "이게 최선이야?"
2. **리서치를 아끼지 마라**: 웹 검색 + Agent 탐색으로 충분히 조사
3. **병렬로 돌려라**: Agent tool로 여러 에이전트 동시 실행
4. **Opus를 써라**: 아키텍처/분석은 항상 Opus
5. **실험 해석을 AI에게 맡겨라**: "이 결과를 분석해줘"
6. **대기 시간을 낭비하지 마라**: 테스트는 background, 대기 중에 다음 작업

## Brain 지식 참조 (자가진화 루프)

Comad Brain(지식 그래프)이 MCP로 연결되어 있을 때, 코딩 시 축적된 지식을 자동 참조한다.

**자동 참조 시점:**
- 세션 시작 시: `comad_brain_trend`로 최근 트렌드 1회 로드
- 새 프로젝트 또는 기술 스택 선택 시: `comad_brain_search`로 경량 조회
- 사용자가 심층 분석을 요청할 때만: `comad_brain_ask` 사용 (무거움)

**규칙:**
- 모든 참조 결과에 수집일 표시 (freshness)
- Brain 조언은 '참고'이며, 사용자 결정을 우선한다
- Brain MCP가 연결되지 않은 환경에서는 이 섹션을 무시한다

**내재화 원칙:**
- 외부 패키지 추가 금지. 패턴/알고리즘만 학습하여 자체 구현
- 아이디어 차용 OK, 의존성 수입 NO
- 인프라 스택(Neo4j, Bun, Python)은 현재 확정된 것만 허용
