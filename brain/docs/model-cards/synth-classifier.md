# Model Card: synth-classifier

## Model Details

- 모델명: `synth-classifier`
- 구현 위치: `brain/packages/graphrag/src/synth-classifier.ts`
- 유형: 규칙 기반 분류기, 외부 모델 호출 없음
- 출력: `easy | hard` 복잡도 티어와 근거 문자열 목록
- 설계 의도: GraphRAG 질의 라우팅에서 값비싼 합성 경로를 언제 사용할지 결정하는 1차 게이트

이 분류기는 이름만 보면 일반 목적 classifier처럼 보이지만, 실제로는 질문 복잡도를 판정해 질의 비용과 응답 경로를 조정하는 운영용 휴리스틱이다. 신뢰도 자체를 예언하는 모델이 아니라 "이 질문이 얼마나 넓은 문맥과 다중 추론을 요구하는가"를 거칠게 구분한다.

## Intended Use

적합한 사용:
- 짧은 사실 질문과 긴 합성 질문을 분리해 라우팅 비용을 통제할 때
- 기술 문서, GraphRAG 문맥, 연구 요약 질의에서 1차 complexity gating이 필요할 때
- 영어와 한국어 질문을 함께 다루되 정밀 의미 해석보다 보수적 분류가 중요할 때

부적합한 사용:
- 사용자 의도 전체를 이해하는 semantic classifier 용도
- 도메인 라벨링, 감성 분석, 안전성 판정 용도
- 결과를 절대적 품질 신호나 사실 신뢰도 점수로 해석하는 용도

즉, `synth-classifier`가 분류하는 것은 기술 도메인 자체가 아니라 질문의 처리 난이도다. 다만 난이도 판단의 근거는 기술적 다중홉 여부, 도메인 폭, 문맥 길이, 설명 요구 강도에 있다.

## Factors

성능에 영향을 주는 주요 요인:
- 질문 길이. 80자를 넘으면 hard로 기운다.
- 키워드. `compare`, `analyze`, `왜`, `비교` 같은 다중 추론 신호가 hard를 유도한다.
- 문장 절 수. 쉼표나 물음표가 많아 다중 절이면 hard 가능성이 높다.
- 컨텍스트 길이. `contextLen > 1200`이면 wide-context로 판정한다.
- 언어 특성. 한국어는 띄어쓰기와 형태소가 다양하므로 exact substring 방식이 누락을 만들 수 있다.

## Metrics

현재 구현은 확률 점수나 calibration을 제공하지 않는다. 운영 메트릭은 다음처럼 보는 것이 적절하다.

- Route Precision: hard로 보낸 질문 중 실제로 고비용 합성이 필요했던 비율
- Route Recall: 실제 hard 질문을 얼마나 놓치지 않았는지
- Cost Avoidance: easy로 처리해 절감한 평균 호출 비용
- Misroute Rate: 잘못된 티어 분류로 인해 품질 저하나 비용 낭비가 발생한 비율

현재 테스트는 경계값 회귀 방지에 초점이 있다. 따라서 높은 분류 정확도를 증명하는 벤치마크가 아니라, 휴리스틱이 의도치 않게 무너지지 않도록 잠그는 수준의 검증이다.

## Training Data

학습 데이터는 없다. 이 분류기는 수작업으로 작성한 키워드 목록, 길이 임계값, 구두점 규칙에 의해 동작한다. 따라서 통계적 일반화 능력보다 운영자 판단이 직접 성능을 좌우한다. "훈련"에 해당하는 것은 사실상 규칙 수정과 회귀 테스트 추가다.

## Evaluation Data

평가 데이터는 `synth-classifier.test.ts`의 예제 질문 세트다. 영어 factual 질문, 비교형 질문, 한국어 추론 질문, 문맥 길이 경계값, 공백 문자열 등의 사례가 포함된다. 하지만 이 데이터는 작고 합성적이므로 실제 사용자 분포를 대표하지 않는다. 특히 비기술 질문, 혼합 언어 질문, 장문의 모호한 질문에 대한 외삽은 제한적이다.

## Ethical Considerations

이 분류기는 낮은 위험처럼 보이지만, 라우팅 게이트라는 점에서 간접적 영향이 있다. 특정 언어권이나 서술 스타일이 hard로 과도하게 분류되면 더 비싼 경로를 타거나, 반대로 중요한 질문이 easy로 축소되어 빈약한 답을 받을 수 있다. 또한 "hard"를 신뢰도 낮음으로 오해하면 사용자 질문의 정당성을 잘못 평가할 수 있다.

## Caveats

- 복잡도 2단계만 제공하므로 중간 난이도를 표현하지 못한다.
- 기술/도메인/신뢰도 분류기는 아니다. 질문 처리 난이도 게이트일 뿐이다.
- 키워드 기반이라 신조어, 완곡표현, 비문형 질문을 놓칠 수 있다.
- `contextLen`이 크면 내용과 무관하게 hard가 되므로 보수 편향이 있다.
- 규칙 기반이라 설명 가능성은 높지만 적응성은 낮다.

결론적으로 `synth-classifier`는 저비용 운영 분기점으로는 적합하지만, 일반 목적 판단기처럼 과신하면 안 된다.
