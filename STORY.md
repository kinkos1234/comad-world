# Comad World — Origin Story

## 시작: 왜 만들었나

매일 아침 arXiv를 읽는 루틴은 흔하다. 흔치 않은 건 **읽은 것이 누적되는 시스템**이다. 노션에 요약하고, 북마크를 쌓고, 괜찮은 스레드를 저장해도, 한 달 뒤 그중 무엇이 내 생각을 바꿨는지 정직하게 답하기 어렵다. 정보는 들어왔지만, **도구는 어제와 같다**.

Comad World는 그 비대칭을 지우려는 시도다. 읽는 행위가 그래프의 엣지 하나를 만들고, 엣지가 렌즈의 가중치를 조정하고, 렌즈가 다음 크롤 우선순위를 역주입한다. 읽으면 도구가 미세하게 다른 도구가 된다 — 유일한 설계 원칙이 그것이다.

## 왜 6개 모듈인가

감각·사고·행동을 분리한 생물학적 메타포를 그대로 코드에 옮긴 결과다. 각 모듈은 **자기 `.git`을 가지고**, `comad.config.yaml` 하나로 묶인다 (ADR 0001).

- **ear** — 듣는 일. RSS / HN / arXiv / GitHub 수신. 단방향 입력 계층.
- **brain** — 연결하는 일. Neo4j 위의 지식 그래프 + MCP 서버. 의미를 저장하는 유일한 곳.
- **eye** — 예측하는 일. 5-lens 시뮬레이션, 예측 정확도 추적. brain의 해석 레이어.
- **photo** — 시각을 다루는 일. Photoshop MCP 연동. 실험적 분기.
- **sleep** — 잊지 않는 일. 세션 간 메모리 통합과 자동 pruning. 시간 축 관리자.
- **voice** — 반복을 자동화하는 일. 재발 워크플로우와 OS-awareness scheduler.

하나를 빼면 루프가 끊긴다. 하나를 더하면 (아직은) 불필요한 커플링을 만든다. 6은 최소 완결 집합에 가깝다.

## 실패담

**첫 번째 — Neo4j 단일 인스턴스의 붕괴.** 초기에는 brain과 eye가 같은 Neo4j를 공유했다. crawling 쓰기와 simulation 쿼리가 경합하면서 lock contention이 잦아졌고, p95 latency가 20.7초까지 올라갔다. "하나면 충분하다"는 직관이 스케일에서 비용으로 돌아왔다. 결국 **7687(eye) / 7688(brain) 분리 + 각 힙 2G 고정**으로 분기한 뒤 13.8초로 내려갔다. 경계는 설계서가 아니라 통증이 가르친다.

**두 번째 — 17K 줄을 지워야 했던 세션.** v0.2.0 직전, 기능을 추가하는 속도보다 **빼야 할 코드가 더 많다**는 사실이 드러났다. 하드코딩된 경로, 중복된 module README, 더 이상 호출되지 않는 어댑터가 모듈마다 쌓여 있었다. `lib/common.sh` 추출, path-agnostic 템플릿, Structure Guard CI(루트에 모듈 파일 중복 시 PR reject)를 도입한 뒤에야 구조가 다시 읽혔다. **과설계는 조용히 쌓이고 시끄럽게 터진다**.

## 지금 배운 것

- 시스템 성능보다 **사용자 변환 서사**가 먼저다 — 기능이 9점이어도 이야기가 6점이면 star는 안 는다.
- 피드백 루프는 **측정되지 않으면 상상일 뿐이다** — 예측 반증, 소스 다양성, temporal decay를 숫자로 남긴다.
- 모듈 경계는 **같이 자라는 것**이지 설계서에서 태어나지 않는다.
